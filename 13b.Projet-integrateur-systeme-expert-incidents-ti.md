<summary><strong>14 — Projet intégrateur — Système expert de priorisation des incidents informatiques</strong></summary>

<br/>

## Objectif du projet

Dans ce projet intégrateur, vous allez concevoir une application capable d’analyser des incidents informatiques à partir de données synthétiques.

L’objectif est de mettre en pratique les notions vues dans le cours :

* les faits ;
* les règles IF-THEN ;
* le moteur d’inférence ;
* l’explication du raisonnement ;
* la priorisation d’une décision ;
* l’utilisation d’une interface simple pour présenter les résultats.

Le projet vise à montrer comment un système expert peut être utilisé dans un contexte professionnel pour soutenir la prise de décision.

---

## Contexte professionnel

Une organisation reçoit chaque jour plusieurs incidents informatiques.

Ces incidents peuvent concerner :

* une application inaccessible ;
* une panne réseau ;
* une erreur de base de données ;
* un problème de sécurité ;
* un problème d’accès utilisateur ;
* un ralentissement de service ;
* une panne touchant plusieurs employés.

Dans un contexte réel, l’équipe de support doit décider rapidement quels incidents traiter en priorité.

Un incident qui touche un seul utilisateur n’a pas nécessairement la même urgence qu’une panne qui bloque un département complet. De la même façon, un problème ayant un impact financier ou un risque de sécurité doit être traité avec plus d’attention.

Le système expert à développer devra donc aider à répondre à la question suivante :

```text
Quelle priorité faut-il attribuer à cet incident informatique ?
```

Les priorités possibles sont :

```text
Faible
Moyenne
Élevée
Critique
```

---

## Problème à résoudre

Vous devez concevoir un système expert capable de :

1. Lire une liste d’incidents informatiques.
2. Analyser les faits associés à chaque incident.
3. Appliquer une base de règles.
4. Calculer une priorité finale.
5. Expliquer les règles déclenchées.
6. Présenter les résultats dans une interface claire.

Le système ne doit pas prendre une décision de manière opaque. Chaque priorité attribuée doit être justifiée par des règles compréhensibles.

---

## Données synthétiques

Les données utilisées dans ce projet seront fictives.

Elles doivent représenter des incidents informatiques réalistes, mais ne doivent contenir aucune donnée réelle d’entreprise, aucun nom de personne et aucune information confidentielle.

Le fichier de données devra s’appeler :

```text
incidents_ti_synthetiques.csv
```

Il devra contenir au minimum 200 incidents.

## Base de règles minimale

Le système expert doit contenir au minimum 12 règles.

Les règles suivantes peuvent être utilisées comme base de départ.

```text
R1 : SI risque_securite = oui
ALORS priorité = Critique

R2 : SI service_bloque = oui ET nombre_utilisateurs_touches >= 20
ALORS priorité = Critique

R3 : SI impact_financier = élevé ET solution_contournement = non
ALORS priorité = Critique

R4 : SI service_bloque = oui ET nombre_utilisateurs_touches entre 5 et 19
ALORS priorité = Élevée

R5 : SI categorie = Base de données ET impact_financier = élevé
ALORS priorité = Élevée

R6 : SI categorie = Réseau ET nombre_utilisateurs_touches >= 10
ALORS priorité = Élevée

R7 : SI anciennete_incident_h > 24
ALORS augmenter la priorité d’un niveau

R8 : SI solution_contournement = oui ET risque_securite = non
ALORS réduire l’urgence de traitement

R9 : SI categorie = Accès utilisateur ET nombre_utilisateurs_touches = 1
ALORS priorité = Faible

R10 : SI categorie = Performance ET nombre_utilisateurs_touches >= 10
ALORS priorité = Moyenne ou Élevée selon l’impact financier

R11 : SI departement = Direction ET service_bloque = oui
ALORS priorité minimale = Élevée

R12 : SI impact_financier = faible ET solution_contournement = oui ET nombre_utilisateurs_touches <= 2
ALORS priorité = Faible
```

---

## Gestion des contradictions

Certaines règles peuvent produire des conclusions différentes.

Exemple :

```text
R1 : SI risque_securite = oui ALORS priorité = Critique
R8 : SI solution_contournement = oui ALORS réduire l’urgence de traitement
```

Dans ce cas, le risque de sécurité doit rester prioritaire.

Le système doit donc prévoir une stratégie de résolution des conflits.

Une méthode simple consiste à utiliser un score.

---

## Système de score recommandé

Chaque incident reçoit un score en fonction des faits observés.

| Fait observé                             | Points |
| ---------------------------------------- | -----: |
| Risque de sécurité                       |    +50 |
| Service bloqué                           |    +30 |
| 20 utilisateurs touchés ou plus          |    +30 |
| Entre 5 et 19 utilisateurs touchés       |    +15 |
| Impact financier élevé                   |    +25 |
| Impact financier moyen                   |    +10 |
| Aucune solution de contournement         |    +20 |
| Incident ouvert depuis plus de 24 heures |    +15 |
| Solution de contournement disponible     |    -10 |
| Un seul utilisateur touché               |    -10 |

La priorité finale est ensuite déterminée selon le score.

| Score final | Priorité |
| ----------- | -------- |
| 0 à 20      | Faible   |
| 21 à 45     | Moyenne  |
| 46 à 75     | Élevée   |
| 76 et plus  | Critique |

## Exemple d’analyse d’un incident

### Faits observés

```text
Département : Finance
Catégorie : Application inaccessible
Nombre d’utilisateurs touchés : 38
Service bloqué : oui
Risque de sécurité : non
Impact financier : élevé
Solution de contournement : non
Ancienneté de l’incident : 2 heures
```

### Règles déclenchées

```text
R2 : service bloqué et nombre d’utilisateurs touchés supérieur ou égal à 20
R3 : impact financier élevé et absence de solution de contournement
```

### Calcul du score

```text
Service bloqué : +30
20 utilisateurs touchés ou plus : +30
Impact financier élevé : +25
Aucune solution de contournement : +20

Score total : 105
```

### Conclusion

```text
Priorité finale : Critique
```

### Explication

```text
L’incident est classé critique, car il bloque un service utilisé par un grand nombre d’utilisateurs, il concerne un département sensible, il présente un impact financier élevé et aucune solution temporaire n’est disponible.
```

## Application à développer

L’application doit être développée en Python.

Une interface avec Streamlit est recommandée afin de présenter les résultats de manière claire.

L’application doit permettre de :

1. Charger le fichier CSV.
2. Afficher les incidents.
3. Appliquer automatiquement le système expert.
4. Ajouter une colonne de priorité calculée.
5. Ajouter une colonne avec les règles déclenchées.
6. Ajouter une colonne avec l’explication de la décision.
7. Filtrer les incidents par département.
8. Filtrer les incidents par catégorie.
9. Filtrer les incidents par priorité.
10. Afficher le nombre d’incidents par priorité.
11. Afficher le nombre d’incidents par département.
12. Sélectionner un incident et consulter son raisonnement détaillé.

## Commandes d’exécution

Créer un environnement virtuel :

```bash
python -m venv .venv
```

Activer l’environnement sous Windows :

```bash
.venv\Scripts\activate
```

Installer les dépendances :

```bash
pip install -r requirements.txt
```

Générer les données synthétiques :

```bash
python generer_donnees.py
```

Lancer l’application :

```bash
streamlit run app.py
```
