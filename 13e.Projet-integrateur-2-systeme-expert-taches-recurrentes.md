# Projet — Système expert de gestion intelligente des tâches récurrentes et des rappels

## 1. Titre du projet

**Développement d’un système expert simple pour la gestion intelligente des tâches récurrentes avec calendrier, backend, base de données et règles de décision**

---

## 2. Contexte du projet

Dans la vie réelle, plusieurs tâches reviennent régulièrement. Certaines doivent être faites chaque mois, chaque semaine ou à une date précise. Par exemple, une personne peut devoir payer une facture d’électricité, payer Internet, payer un loyer, renouveler une assurance, envoyer un rapport, faire une sauvegarde de fichiers, vérifier un abonnement ou effectuer une tâche administrative importante.

Ces tâches semblent simples, mais elles peuvent facilement être oubliées, surtout lorsqu’elles reviennent souvent. Lorsqu’une tâche n’est pas faite à temps, elle devient en retard. Cela peut créer des problèmes : frais supplémentaires, oubli d’un paiement, retard dans un travail, perte d’information ou mauvaise organisation.

Dans ce projet, vous devez développer une application qui aide l’utilisateur à mieux gérer ses tâches récurrentes. Chaque étudiant ou chaque équipe peut choisir ses propres exemples de tâches selon le contexte souhaité. L’important est que les tâches soient récurrentes, enregistrées dans une base de données et affichées clairement dans un calendrier.

L’application devra permettre à l’utilisateur d’ajouter une tâche principale, par exemple :

```text
Payer la facture mensuelle d’électricité chaque 15 du mois.
```

À partir de cette tâche principale, le backend devra générer automatiquement les rappels correspondants pour plusieurs mois. Ces rappels devront ensuite être enregistrés dans la base de données et affichés dans un calendrier.

Le projet doit être présenté comme un système expert simple, car l’application devra appliquer des règles de décision de type `IF / ELSE`. Ces règles permettront au système d’analyser chaque tâche et de décider automatiquement de son état.

Exemple :

```text
IF la date de la tâche est dépassée
AND la tâche n’est pas complétée
THEN la tâche devient en retard
AND elle s’affiche en rouge
```

Autre exemple :

```text
IF la tâche est complétée
THEN la tâche devient terminée
AND elle s’affiche en vert
```

Ainsi, l’application ne se limite pas à afficher une simple liste de tâches. Elle doit être capable de raisonner automatiquement à partir des dates, des statuts et des règles définies dans le backend.

L’objectif du projet est donc de construire une application complète qui combine :

* une interface utilisateur claire ;
* un calendrier visuel ;
* un backend ;
* une base de données ;
* des opérations CRUD ;
* une génération automatique des rappels ;
* des règles `IF / ELSE` ;
* une coloration automatique des tâches selon leur état.

Chaque étudiant doit choisir ses propres exemples de tâches. Le projet peut donc être adapté à différents contextes : personnel, scolaire, professionnel, administratif, financier ou organisationnel.

---

## 3. Objectif général du projet

L’objectif du projet est de développer une application web permettant de créer, consulter, modifier, supprimer et suivre des tâches récurrentes à partir d’un calendrier intelligent.

L’application doit permettre à l’utilisateur de définir une tâche récurrente, par exemple :

```text
Payer la facture mensuelle d’électricité chaque 15 du mois
du 15 juin 2026 au 15 juin 2027.
```

Le backend doit ensuite générer automatiquement toutes les tâches mensuelles correspondantes et les enregistrer dans la base de données.

Le système doit également appliquer automatiquement des règles pour déterminer si chaque tâche est :

* normale ;
* urgente ;
* en retard ;
* complétée ;
* annulée ;
* archivée.

## 4. Idée principale du projet

Le frontend ne doit pas seulement afficher des tâches créées manuellement.

Le cœur du projet doit être dans le backend.

Le backend doit être responsable de :

* recevoir les informations saisies par l’utilisateur ;
* valider les données ;
* créer une tâche récurrente principale ;
* générer automatiquement les rappels associés ;
* sauvegarder les rappels dans la base de données ;
* appliquer les règles du système expert ;
* retourner au frontend les tâches avec leur état calculé ;
* fournir au frontend la couleur à afficher dans le calendrier.

Le frontend doit principalement servir à :

* afficher les données ;
* permettre la saisie ;
* afficher le calendrier ;
* appeler l’API du backend ;
* afficher les résultats calculés par le backend.

Le frontend ne doit pas décider seul si une tâche est en retard, urgente ou complétée. Cette décision doit être calculée dans le backend à l’aide des règles du système expert.

## 7. Nature du système expert

Ce projet n’est pas un système expert complexe avec intelligence artificielle avancée.

Il s’agit d’un système expert simple à base de règles.

Un système expert à base de règles prend des décisions à partir de conditions.

Dans ce projet, les règles seront écrites avec des conditions simples :

```text
IF condition
THEN décision
```

ou :

```text
IF condition
AND autre condition
THEN action
```

Le système doit analyser les tâches et décider automatiquement de leur état.

---

## 8. Règles obligatoires du système expert

Le backend doit contenir un moteur de règles.

Ce moteur doit analyser les tâches enregistrées dans la base de données.

### Règle 1 — Tâche en retard

```text
IF date_de_la_tâche < date_actuelle
AND statut != "complétée"
THEN état_expert = "en retard"
AND couleur = "rouge"
```

Explication :

Si la date prévue est déjà passée et que l’utilisateur n’a pas marqué la tâche comme complétée, la tâche doit être considérée comme en retard.

---

### Règle 2 — Tâche urgente

```text
IF date_de_la_tâche >= date_actuelle
AND date_de_la_tâche <= date_actuelle + 7 jours
AND statut != "complétée"
THEN état_expert = "urgent"
AND couleur = "orange"
```

Explication :

Si une tâche arrive dans les sept prochains jours, elle doit être affichée comme urgente.

---

### Règle 3 — Tâche normale

```text
IF date_de_la_tâche > date_actuelle + 7 jours
AND statut != "complétée"
THEN état_expert = "normal"
AND couleur = "bleu"
```

Explication :

Si la tâche est encore loin dans le futur, elle est normale.

---

### Règle 4 — Tâche complétée

```text
IF statut == "complétée"
THEN état_expert = "terminée"
AND couleur = "vert"
```

Explication :

Si l’utilisateur a marqué la tâche comme complétée, elle doit rester en vert, même si la date est passée.

---

### Règle 5 — Tâche annulée

```text
IF statut == "annulée"
THEN état_expert = "annulée"
AND couleur = "gris"
```

Explication :

Une tâche annulée ne doit plus être considérée comme urgente ou en retard.

---

### Règle 6 — Tâche archivée

```text
IF statut == "archivée"
THEN état_expert = "archivée"
AND couleur = "gris"
```

Explication :

Une tâche archivée doit rester dans l’historique, mais elle ne doit plus apparaître comme une tâche active.

## 9. Fonctionnalités obligatoires

### 9.1 Création d’une tâche récurrente principale

L’utilisateur doit pouvoir créer une tâche récurrente à partir d’un formulaire.

Le formulaire doit contenir au minimum :

```text
Titre
Description
Montant ou valeur associée
Date de début
Date de fin
Jour du mois
Fréquence
Statut
```

Exemple :

```text
Titre : Payer la facture mensuelle d’électricité
Description : Paiement obligatoire chaque mois.
Montant : 120 $
Date de début : 2026-06-15
Date de fin : 2027-06-15
Jour du mois : 15
Fréquence : mensuelle
Statut : active
```

Après la soumission du formulaire, le frontend doit envoyer les données au backend.

Le backend doit créer la tâche principale et générer les rappels associés.

---

### 9.2 Génération automatique des rappels par le backend

Le backend ne doit pas seulement enregistrer une seule tâche.

Il doit générer toutes les tâches comprises entre la date de début et la date de fin.

Exemple :

```text
Date de début : 15 juin 2026
Date de fin : 15 juin 2027
Fréquence : mensuelle
```

Le backend doit générer 13 rappels.

Chaque rappel doit être sauvegardé dans la table des tâches ou des rappels.

---

### 9.3 Affichage dans un calendrier

L’application doit afficher un calendrier clair.

Le calendrier doit permettre de voir :

* les tâches du mois ;
* les tâches passées ;
* les tâches futures ;
* les tâches urgentes ;
* les tâches en retard ;
* les tâches complétées ;
* les tâches annulées ;
* les tâches archivées.

Chaque tâche doit être affichée avec une couleur selon son état.

Couleurs minimales attendues :

```text
Rouge  : en retard
Orange : urgent
Vert   : complétée
Bleu   : normale
Gris   : annulée ou archivée
```

---

### 9.4 Modification d’une tâche

L’utilisateur doit pouvoir modifier une tâche.

Il peut modifier par exemple :

```text
Le titre
La description
Le montant ou la valeur associée
La date
Le statut
```

Attention : si une tâche est modifiée, le backend doit réappliquer les règles du système expert.

---

### 9.5 Modification d’une tâche récurrente principale

L’utilisateur doit pouvoir modifier une tâche récurrente principale.

Par exemple, il peut changer :

```text
Le montant
La date de fin
Le jour du mois
Le titre
La description
La fréquence
```

Le système doit gérer correctement les tâches déjà générées.

Deux approches sont acceptées :

### Approche 1 — Régénération complète

Le backend supprime les anciennes tâches non complétées et génère une nouvelle série.

### Approche 2 — Mise à jour partielle

Le backend conserve les tâches déjà complétées et met à jour seulement les tâches futures.

Les étudiants doivent expliquer l’approche choisie dans leur README.

---

### 9.6 Suppression d’une tâche individuelle

L’utilisateur doit pouvoir supprimer une tâche individuelle.

Exemple :

```text
Supprimer uniquement la tâche du 15 août 2026.
```

La suppression doit être sauvegardée dans la base de données.

---

### 9.7 Suppression ou archivage d’une tâche récurrente principale

L’utilisateur doit pouvoir supprimer ou archiver une tâche récurrente principale.

Exemple :

```text
Une ancienne série de rappels est terminée.
L’utilisateur veut l’archiver pour ne plus l’afficher dans les tâches actives.
```

Si une tâche récurrente principale est supprimée, le backend doit gérer les rappels associés.

Deux options sont possibles :

```text
Option 1 : supprimer tous les rappels associés.
Option 2 : archiver la tâche principale et ses rappels.
```

Le choix doit être expliqué dans le README.

---

### 9.8 Marquer une tâche comme complétée

L’utilisateur doit pouvoir cliquer sur une tâche et la marquer comme complétée.

Après cette action :

```text
statut = "complétée"
état_expert = "terminée"
couleur = "vert"
```

La modification doit être enregistrée dans la base de données.

---

### 9.9 Liste des tâches en retard

L’application doit afficher une section spéciale :

```text
Tâches en retard
```

Cette section doit lister uniquement les tâches dont l’état expert est :

```text
en retard
```

---

### 9.10 Liste des tâches urgentes

L’application doit afficher une section spéciale :

```text
Tâches urgentes
```

Cette section doit lister uniquement les tâches qui arrivent bientôt.

---

### 13.1 Routes pour les tâches récurrentes principales

```text
GET /recurring-tasks
```

Retourne toutes les tâches récurrentes principales.

```text
GET /recurring-tasks/:id
```

Retourne une tâche récurrente précise.

```text
POST /recurring-tasks
```


Modifie une tâche récurrente principale.

### 13.2 Routes pour les rappels


Retourne tous les rappels.

```text

```


```text

```

```text

```

```text

Retourner les rappels mis à jour
```

## 15. Logique backend attendue

Le backend doit contenir au minimum trois parties importantes.

---

### 15.1 Service de génération des rappels

Ce service reçoit une tâche récurrente principale et génère les rappels mensuels.

Pseudo-code :

```text
recevoir date_debut
recevoir date_fin
recevoir jour_du_mois

date_courante = date_debut

WHILE date_courante <= date_fin
    créer un rappel avec date_courante
    ajouter un mois à date_courante
END WHILE
```

---

### 15.2 Service du système expert

Ce service applique les règles.

Pseudo-code :

```text
POUR chaque rappel dans la base de données

    IF statut == "complétée"
        état = "terminée"
        couleur = "vert"

    ELSE IF statut == "annulée"
        état = "annulée"
        couleur = "gris"

    ELSE IF statut == "archivée"
        état = "archivée"
        couleur = "gris"

    ELSE IF date_rappel < date_actuelle
        état = "en retard"
        couleur = "rouge"

    ELSE IF date_rappel <= date_actuelle + 7 jours
        état = "urgent"
        couleur = "orange"

    ELSE
        état = "normal"
        couleur = "bleu"

    sauvegarder état et couleur

FIN POUR
```

---

### 15.3 Service CRUD

Ce service permet de :

```text
Créer
Lire
Modifier
Supprimer
```

les tâches récurrentes principales et les rappels.

Le frontend envoie les données au backend.

### Étape 5

Le backend crée la tâche récurrente principale dans la base de données.

### Étape 6

Le backend génère automatiquement tous les rappels mensuels.

### Étape 7

Le backend applique les règles du système expert.

### Étape 8

Le frontend affiche les rappels dans le calendrier.

### Étape 9

Si un rappel est en retard, il apparaît en rouge.

### Étape 10

L’utilisateur peut cliquer sur un rappel et le marquer comme complété.

## 33. Conclusion

Ce projet vous permet de développer une application complète qui ressemble à un vrai outil professionnel.

Même si l’application semble être un simple calendrier de tâches, elle contient une logique de système expert, car elle prend automatiquement des décisions à partir de règles.

Le backend joue un rôle central : il génère les rappels, applique les règles, sauvegarde les données et fournit les résultats au frontend.

Le projet permet donc de pratiquer plusieurs compétences importantes :

```text
Développement frontend
Développement backend
Base de données
API REST
CRUD
Règles IF / ELSE
Calendrier interactif
Architecture logicielle
Documentation technique
Présentation professionnelle
```

Votre objectif est de livrer une application claire, fonctionnelle, bien structurée et capable de démontrer une vraie logique de décision.
