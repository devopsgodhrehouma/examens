# 11a — Tutoriel DAX dans Power BI expliqué simplement

## Objectif du tutoriel

Ce tutoriel a pour objectif d’expliquer DAX dans Power BI de manière progressive, simple et très concrète.

L’idée n’est pas seulement de donner des formules DAX à mémoriser. L’objectif est surtout de comprendre pourquoi DAX existe, à quel moment il devient nécessaire, ce qu’il permet de faire dans un rapport Power BI, et pourquoi un rapport professionnel devient rapidement limité si on ne connaît pas au moins les bases de DAX.

À la fin de ce tutoriel, vous devez être capable de comprendre la différence entre afficher des données et analyser des données. Vous devez aussi être capable d’expliquer le rôle d’une mesure, d’une colonne calculée, du contexte de filtre, du contexte de ligne, et des fonctions DAX les plus utilisées dans les premiers rapports Power BI.

<summary>1 - Pourquoi parler de DAX dans Power BI ?</summary>

<br/>

Quand on commence avec Power BI, on pense souvent que l’outil sert seulement à créer des graphiques.

On importe un fichier Excel, on glisse une colonne dans un graphique, on ajoute un filtre, et Power BI affiche automatiquement des résultats. Cette première impression est normale, car Power BI est effectivement très puissant pour créer rapidement des visualisations.

Cependant, dans un contexte professionnel, un rapport Power BI ne sert pas uniquement à montrer des chiffres. Il sert surtout à aider une personne, une équipe ou une organisation à prendre une décision.

Par exemple, une direction ne veut pas seulement voir une colonne contenant les ventes. Elle veut comprendre si les ventes sont bonnes, si elles augmentent, si elles diminuent, si elles dépassent les objectifs, si une région performe mieux qu’une autre, si un produit est rentable, ou si un client rapporte suffisamment par rapport aux coûts associés.

C’est exactement ici que DAX devient important.

DAX permet de transformer des données simples en indicateurs utiles.

Sans DAX, Power BI peut afficher des données.

Avec DAX, Power BI peut analyser les données.

La différence est très importante.

Afficher une donnée, c’est montrer ce qui existe déjà dans une table.

Analyser une donnée, c’est créer une information nouvelle à partir des données existantes.

Par exemple, une table peut contenir une colonne `Quantité` et une colonne `PrixUnitaire`.

Ces deux colonnes sont des données brutes.

Mais le chiffre d’affaires n’existe pas forcément comme colonne dans la table.

Pour obtenir le chiffre d’affaires, il faut faire un calcul :

```text
Chiffre d’affaires = Quantité × Prix unitaire
````

Dans Power BI, ce type de calcul peut être créé avec DAX.

C’est pour cela que DAX est une compétence essentielle dès que l’on veut créer des rapports sérieux, professionnels et réellement utiles.

</details>

<details>
<summary>2 - Power BI sans DAX : ce qu’on peut déjà faire</summary>

<br/>

Avant de dire que DAX est nécessaire, il faut être juste : Power BI peut déjà faire beaucoup de choses sans que l’utilisateur écrive une seule formule DAX.

Power BI peut importer des données provenant de plusieurs sources :

* fichiers Excel ;
* fichiers CSV ;
* bases de données SQL ;
* dossiers ;
* services cloud ;
* SharePoint ;
* API ;
* plateformes analytiques.

Power BI permet aussi de préparer les données avec Power Query. On peut supprimer des colonnes, changer des types de données, renommer des champs, filtrer des lignes, fusionner des tables, corriger des erreurs et préparer les données avant de les utiliser dans un rapport.

Ensuite, Power BI permet de créer des visualisations :

* graphiques en barres ;
* graphiques en lignes ;
* cartes ;
* tableaux ;
* matrices ;
* segments ;
* indicateurs ;
* filtres interactifs.

Il peut également faire automatiquement certains calculs simples.

Par exemple, si une table contient une colonne `Montant`, Power BI peut automatiquement afficher :

```text
Somme de Montant
```

ou :

```text
Moyenne de Montant
```

ou encore :

```text
Nombre de lignes
```

Cela veut dire qu’au départ, on peut créer un rapport simple sans DAX.

Par exemple, un rapport très simple peut répondre à des questions comme :

* Quel est le total des ventes ?
* Combien de lignes contient la table ?
* Quelle est la moyenne des montants ?
* Combien de produits sont listés ?
* Quelles sont les ventes par région ?
* Quelles sont les ventes par catégorie ?

Mais ce type de rapport reste limité.

Il montre principalement ce qui est déjà directement présent dans les données.

Il ne permet pas encore de construire des indicateurs avancés, des comparaisons intelligentes, des pourcentages dynamiques ou des calculs métier spécifiques.

C’est à ce moment-là que DAX devient nécessaire.

</details>

<details>
<summary>3 - Le vrai problème : afficher des données ne suffit pas</summary>

<br/>

Dans un rapport professionnel, afficher des données ne suffit pas.

Une donnée brute est une information de départ. Elle est utile, mais elle ne donne pas toujours une réponse directe à une question métier.

Prenons un exemple très simple.

Une entreprise possède une table de ventes avec les colonnes suivantes :

| Produit | Quantité | Prix unitaire | Coût unitaire |
| ------- | -------: | ------------: | ------------: |
| Souris  |        3 |            20 |            10 |
| Clavier |        2 |            50 |            30 |
| Écran   |        1 |           200 |           150 |

Ces données sont utiles.

Mais elles ne répondent pas directement aux questions suivantes :

* Quel est le chiffre d’affaires total ?
* Quel est le coût total ?
* Quelle est la marge ?
* Quel est le taux de marge ?
* Quel produit est le plus rentable ?
* Quel produit génère le plus de revenus ?
* Quelle part représente chaque produit dans le total ?
* Est-ce que les ventes sont meilleures ce mois-ci que le mois dernier ?

Pour répondre à ces questions, il faut créer des calculs.

Ces calculs ne sont pas toujours présents dans les données originales.

C’est exactement le rôle de DAX.

DAX permet de passer de la donnée brute à l’indicateur métier.

### Exemple de différence

Une donnée brute :

```text
Quantité = 3
Prix unitaire = 20
```

Un indicateur calculé :

```text
Chiffre d’affaires = 3 × 20 = 60
```

Une autre donnée brute :

```text
Chiffre d’affaires = 60
Coût total = 30
```

Un autre indicateur calculé :

```text
Marge = 60 - 30 = 30
```

Encore un indicateur :

```text
Taux de marge = 30 / 60 = 50 %
```

On voit donc que les données brutes ne suffisent pas toujours.

Les données brutes donnent les ingrédients.

DAX permet de préparer le plat.

Power BI affiche ensuite le résultat de manière visuelle.

</details>

<a id="section-4"></a>

<details>
<summary>4 - DAX, c’est quoi exactement ?</summary>

<br/>

DAX signifie :

```text
Data Analysis Expressions
```

En français, on peut traduire cela par :

```text
Expressions d’analyse de données
```

DAX est le langage de calcul utilisé dans Power BI.

Il est aussi utilisé dans d’autres outils Microsoft liés à l’analyse de données, comme Analysis Services et Power Pivot dans Excel.

Dans Power BI, DAX sert principalement à créer des calculs qui vont enrichir le rapport.

Ces calculs peuvent être simples ou avancés.

Un calcul simple peut être :

```DAX
TotalVentes = SUM(Ventes[Montant])
```

Un calcul un peu plus avancé peut être :

```DAX
TauxMarge = DIVIDE([Marge], [ChiffreAffaires])
```

Un calcul encore plus avancé peut comparer une période avec une autre :

```DAX
VentesAnneePrecedente =
CALCULATE(
    [TotalVentes],
    SAMEPERIODLASTYEAR(Date[Date])
)
```

L’idée principale est la suivante :

DAX permet de créer des valeurs qui n’existent pas directement dans les données originales.

### Vulgarisation simple

Imagine que Power BI est une cuisine.

Les données sont les ingrédients.

Power Query sert à laver, couper, nettoyer et préparer les ingrédients.

Le modèle de données sert à organiser les ingrédients dans les bons tiroirs et les bonnes tables.

DAX sert à cuisiner.

Les visualisations servent à présenter le plat final.

Sans DAX, tu peux montrer les ingrédients.

Avec DAX, tu peux préparer une vraie recette analytique.

### Autre vulgarisation

Imagine que ton fichier Excel ou ta base de données contient des pièces de Lego.

Power BI peut afficher les pièces.

Power Query peut trier les pièces par couleur et par forme.

Le modèle de données peut organiser les pièces dans des boîtes.

DAX permet de construire quelque chose avec ces pièces.

Par exemple, DAX permet de construire :

* un chiffre d’affaires ;
* une marge ;
* un taux de croissance ;
* un pourcentage du total ;
* un classement ;
* un indicateur de performance ;
* un écart entre réel et objectif.

Donc DAX n’est pas seulement un langage technique.

C’est le langage qui permet de donner du sens aux données.

</details>

<details>
<summary>5 - Pourquoi DAX devient indispensable dans les vrais rapports</summary>

<br/>

DAX devient indispensable quand on veut créer des rapports qui vont au-delà de l’affichage simple.

Dans un vrai rapport d’entreprise, on veut souvent répondre à des questions qui demandent des calculs personnalisés.

Par exemple :

* Quel est le chiffre d’affaires total ?
* Quelle est la marge nette ?
* Quel est le taux de marge ?
* Quelle est la croissance par rapport à l’année précédente ?
* Quel est l’écart entre le budget prévu et le résultat réel ?
* Quel est le pourcentage de contribution de chaque catégorie ?
* Quel est le meilleur produit ?
* Quel est le client le plus important ?
* Combien de clients sont actifs ?
* Quel est le panier moyen ?
* Quel est le taux de conversion ?
* Quelle est la moyenne mobile des ventes ?
* Quels produits dépassent l’objectif ?
* Quelles régions sont en dessous de la cible ?

Ces questions ne peuvent pas toujours être répondues uniquement avec les champs bruts d’une table.

Elles nécessitent des mesures calculées.

### Exemple concret

Supposons que l’entreprise possède cette table :

| Produit   | Ventes | Coûts |
| --------- | -----: | ----: |
| Produit A | 10 000 | 7 000 |
| Produit B |  5 000 | 2 000 |
| Produit C |  8 000 | 6 500 |

Power BI peut afficher les ventes.

Power BI peut afficher les coûts.

Mais l’indicateur vraiment intéressant est souvent la marge.

```text
Marge = Ventes - Coûts
```

En DAX :

```DAX
Marge = [TotalVentes] - [TotalCouts]
```

Ensuite, on peut calculer le taux de marge :

```DAX
TauxMarge =
DIVIDE(
    [Marge],
    [TotalVentes]
)
```

Ce taux est beaucoup plus parlant qu’un simple montant.

Un produit peut avoir beaucoup de ventes, mais une faible marge.

Un autre produit peut avoir moins de ventes, mais une meilleure rentabilité.

Sans DAX, on risque de regarder uniquement les ventes et de prendre une mauvaise décision.

Avec DAX, on peut analyser la rentabilité réelle.

### Idée importante

Power BI sans DAX permet de voir ce qui se passe.

Power BI avec DAX permet de comprendre pourquoi c’est important.

</details>

<details>
<summary>6 - Différence entre Power Query, modèle de données et DAX</summary>

<br/>

Il est très important de ne pas confondre Power Query et DAX.

Les deux servent à travailler avec les données, mais ils ne jouent pas le même rôle.

### Power Query

Power Query sert principalement à préparer les données avant leur chargement dans Power BI.

Il permet de faire des opérations comme :

* supprimer les colonnes inutiles ;
* supprimer les lignes vides ;
* corriger les types de données ;
* transformer du texte ;
* fusionner des tables ;
* ajouter des colonnes de préparation ;
* remplacer des valeurs ;
* nettoyer des erreurs ;
* transformer des dates ;
* structurer les données.

Power Query travaille surtout avant l’analyse.

Il prépare la matière première.

### Le modèle de données

Le modèle de données sert à organiser les tables entre elles.

Par exemple, on peut avoir :

* une table `Ventes` ;
* une table `Produits` ;
* une table `Clients` ;
* une table `Date`.

Le modèle de données permet de créer des relations entre ces tables.

Par exemple :

```text
Ventes[ProduitID] → Produits[ProduitID]
```

ou :

```text
Ventes[ClientID] → Clients[ClientID]
```

Grâce à ces relations, Power BI peut comprendre que chaque vente est associée à un produit, à un client et à une date.

### DAX

DAX sert à faire des calculs analytiques sur le modèle de données.

DAX utilise les tables, les colonnes, les relations et les filtres pour produire des résultats.

### Comparaison simple

| Élément           | Question principale                | Exemple                                    |
| ----------------- | ---------------------------------- | ------------------------------------------ |
| Power Query       | Comment préparer les données ?     | Nettoyer une colonne de dates              |
| Modèle de données | Comment organiser les tables ?     | Relier Ventes et Produits                  |
| DAX               | Comment calculer les indicateurs ? | Calculer la marge ou le taux de croissance |
| Visualisations    | Comment afficher les résultats ?   | Graphique, carte, tableau                  |

### Vulgarisation

Power Query, c’est la préparation avant le match.

Le modèle de données, c’est l’organisation de l’équipe sur le terrain.

DAX, c’est la stratégie de jeu.

Les visualisations, c’est le tableau d’affichage que tout le monde regarde.

</details>

<details>
<summary>7 - Mesure, colonne calculée et table calculée</summary>

<br/>

Dans Power BI, DAX peut être utilisé principalement sous trois formes :

* mesure ;
* colonne calculée ;
* table calculée.

Ces trois éléments utilisent DAX, mais ils ne servent pas exactement à la même chose.

---

## 7.1 La mesure

Une mesure est un calcul dynamique.

Cela veut dire que son résultat change selon les filtres, les graphiques, les segments et le contexte du rapport.

Exemple :

```DAX
TotalVentes = SUM(Ventes[Montant])
```

Cette mesure calcule le total des ventes.

Mais son résultat n’est pas toujours le même.

Si aucun filtre n’est appliqué, elle peut afficher toutes les ventes.

Si on filtre sur l’année 2025, elle affiche seulement les ventes de 2025.

Si on filtre sur Montréal, elle affiche seulement les ventes de Montréal.

Si on filtre sur le produit “Clavier”, elle affiche seulement les ventes du clavier.

La formule est la même, mais le résultat change selon le contexte.

### Vulgarisation

Une mesure est comme une calculatrice intelligente placée dans le rapport.

Elle ne calcule pas dans le vide.

Elle regarde d’abord ce que l’utilisateur a sélectionné, puis elle calcule.

C’est pour cela que les mesures sont très puissantes dans Power BI.

---

## 7.2 La colonne calculée

Une colonne calculée ajoute une nouvelle colonne dans une table.

Elle est calculée ligne par ligne.

Exemple :

```DAX
MontantLigne = Ventes[Quantité] * Ventes[PrixUnitaire]
```

Si la table contient trois lignes, Power BI calcule un résultat pour chaque ligne.

| Produit | Quantité | PrixUnitaire | MontantLigne |
| ------- | -------: | -----------: | -----------: |
| Souris  |        3 |           20 |           60 |
| Clavier |        2 |           50 |          100 |
| Écran   |        1 |          200 |          200 |

La colonne calculée ressemble beaucoup à une colonne ajoutée dans Excel.

Chaque ligne reçoit sa propre valeur.

### Vulgarisation

Une colonne calculée, c’est comme dire :

> Pour chaque ligne de mon tableau, ajoute une nouvelle information.

Elle est utile quand on veut créer une information fixe au niveau de chaque ligne.

---

## 7.3 La table calculée

Une table calculée crée une nouvelle table à partir d’une formule DAX.

Exemple :

```DAX
ProduitsRentables =
FILTER(
    Produits,
    Produits[Marge] > 0
)
```

Les tables calculées sont utiles dans certains cas plus avancés, mais elles ne sont pas la priorité pour un débutant.

Pour commencer, il faut surtout comprendre les mesures et les colonnes calculées.

---

## 7.4 Comment choisir entre mesure et colonne calculée ?

Cette question est très importante.

Il faut se poser la question suivante :

```text
Est-ce que mon résultat doit changer selon les filtres du rapport ?
```

Si la réponse est oui, il faut généralement créer une mesure.

Exemple :

```DAX
TotalVentes = SUM(Ventes[Montant])
```

Cette mesure doit changer selon l’année, la région, le produit ou le client sélectionné.

Maintenant, autre question :

```text
Est-ce que je veux créer une valeur pour chaque ligne de ma table ?
```

Si la réponse est oui, une colonne calculée peut être appropriée.

Exemple :

```DAX
MontantLigne = Ventes[Quantité] * Ventes[PrixUnitaire]
```

### Résumé simple

| Situation                                     | À utiliser       |
| --------------------------------------------- | ---------------- |
| Le résultat doit changer selon les filtres    | Mesure           |
| Le résultat doit être calculé ligne par ligne | Colonne calculée |
| On veut créer une nouvelle table              | Table calculée   |

### Règle pratique

Dans Power BI, on privilégie souvent les mesures pour les indicateurs d’analyse.

Les colonnes calculées doivent être utilisées quand on a vraiment besoin d’une information ligne par ligne dans le modèle.

</details>

<details>
<summary>8 - La notion essentielle : le contexte en DAX</summary>

<br/>

Le contexte est probablement la notion la plus importante en DAX.

On peut apprendre plusieurs fonctions DAX, mais si on ne comprend pas le contexte, on risque de ne pas comprendre pourquoi une formule donne un résultat plutôt qu’un autre.

En DAX, une formule ne calcule jamais complètement seule.

Elle calcule toujours dans un contexte.

Il existe principalement deux notions importantes pour commencer :

* le contexte de filtre ;
* le contexte de ligne.

---

## 8.1 Le contexte de filtre

Le contexte de filtre correspond aux filtres actifs au moment où la mesure est calculée.

Ces filtres peuvent venir de plusieurs endroits :

* un segment dans le rapport ;
* une sélection dans un graphique ;
* une ligne dans un tableau ;
* une colonne utilisée dans une matrice ;
* une relation entre deux tables ;
* un filtre de page ;
* un filtre de rapport ;
* une formule DAX.

Prenons cette mesure :

```DAX
TotalVentes = SUM(Ventes[Montant])
```

Si on l’affiche dans une carte, sans aucun filtre, elle peut donner :

```text
100 000
```

Mais si on affiche la même mesure dans un tableau par région :

| Région   | TotalVentes |
| -------- | ----------: |
| Montréal |      40 000 |
| Québec   |      35 000 |
| Laval    |      25 000 |

La formule DAX n’a pas changé.

C’est toujours :

```DAX
TotalVentes = SUM(Ventes[Montant])
```

Mais le résultat change parce que chaque ligne du tableau applique un filtre différent.

La ligne Montréal filtre les ventes de Montréal.

La ligne Québec filtre les ventes de Québec.

La ligne Laval filtre les ventes de Laval.

### Vulgarisation

Imagine que DAX porte des lunettes.

Quand les lunettes montrent toutes les données, DAX calcule le total général.

Quand les lunettes montrent seulement Montréal, DAX calcule Montréal.

Quand les lunettes montrent seulement 2025, DAX calcule 2025.

Quand les lunettes montrent seulement les produits informatiques, DAX calcule seulement ces produits.

La formule ne change pas.

Ce sont les lunettes qui changent.

Ces lunettes représentent le contexte de filtre.

---

## 8.2 Le contexte de ligne

Le contexte de ligne apparaît quand DAX calcule ligne par ligne.

C’est le cas dans une colonne calculée.

Exemple :

```DAX
MontantLigne = Ventes[Quantité] * Ventes[PrixUnitaire]
```

DAX prend la première ligne, lit la quantité et le prix unitaire, puis calcule le montant.

Ensuite, il passe à la deuxième ligne.

Puis à la troisième.

Chaque ligne a son propre calcul.

| Quantité | PrixUnitaire | MontantLigne |
| -------: | -----------: | -----------: |
|        3 |           20 |           60 |
|        2 |           50 |          100 |
|        1 |          200 |          200 |

### Vulgarisation

Le contexte de ligne, c’est comme un employé qui parcourt un tableau Excel avec son doigt.

Il lit une ligne, fait le calcul, écrit le résultat.

Puis il descend à la ligne suivante.

Il recommence.

C’est une logique ligne par ligne.

---

## 8.3 Pourquoi le contexte est si important ?

Le contexte explique pourquoi une même mesure peut donner plusieurs résultats différents selon l’endroit où elle est placée.

C’est ce qui rend Power BI interactif.

Un utilisateur peut cliquer sur une année, une région, un produit ou une catégorie, et les mesures se recalculent automatiquement.

Sans cette notion, Power BI serait beaucoup moins puissant.

Avec le contexte, une même mesure peut être utilisée dans plusieurs visualisations différentes.

Exemple :

```DAX
TotalVentes = SUM(Ventes[Montant])
```

Cette seule mesure peut servir à afficher :

* le total global des ventes ;
* les ventes par année ;
* les ventes par mois ;
* les ventes par région ;
* les ventes par produit ;
* les ventes par client ;
* les ventes par catégorie.

C’est pour cela que les mesures sont plus puissantes que de simples calculs statiques.

</details>

<a id="section-9"></a>

<details>
<summary>9 - Les fonctions DAX de base à connaître</summary>

<br/>

Il existe beaucoup de fonctions DAX.

Mais au début, il n’est pas nécessaire de tout apprendre.

Il faut surtout maîtriser quelques fonctions essentielles qui reviennent très souvent dans les rapports Power BI.

---

## 9.1 SUM

La fonction `SUM` additionne les valeurs d’une colonne.

Exemple :

```DAX
TotalVentes = SUM(Ventes[Montant])
```

Cette mesure additionne tous les montants de la colonne `Montant`.

Elle est utilisée quand la colonne contient déjà les valeurs à additionner.

### Exemple simple

| VenteID | Montant |
| ------- | ------: |
| 1       |     100 |
| 2       |     200 |
| 3       |      50 |

Résultat :

```text
TotalVentes = 350
```

---

## 9.2 AVERAGE

La fonction `AVERAGE` calcule la moyenne d’une colonne.

Exemple :

```DAX
MoyenneVentes = AVERAGE(Ventes[Montant])
```

Elle est utile quand on veut connaître une valeur moyenne.

### Exemple simple

| VenteID | Montant |
| ------- | ------: |
| 1       |     100 |
| 2       |     200 |
| 3       |     300 |

Résultat :

```text
MoyenneVentes = 200
```

---

## 9.3 COUNT

La fonction `COUNT` compte les valeurs numériques d’une colonne.

Exemple :

```DAX
NombreVentes = COUNT(Ventes[Montant])
```

Elle peut servir à compter le nombre de ventes qui possèdent un montant.

---

## 9.4 DISTINCTCOUNT

La fonction `DISTINCTCOUNT` compte les valeurs uniques.

Exemple :

```DAX
NombreClients = DISTINCTCOUNT(Ventes[ClientID])
```

Cette fonction est très importante.

Si un même client achète plusieurs fois, il ne faut pas forcément le compter plusieurs fois.

Exemple :

| VenteID | ClientID |
| ------- | -------- |
| 1       | C001     |
| 2       | C001     |
| 3       | C002     |
| 4       | C003     |

Le nombre de lignes est 4.

Mais le nombre de clients distincts est 3.

```text
C001, C002, C003
```

Donc :

```DAX
NombreClients = DISTINCTCOUNT(Ventes[ClientID])
```

donnera :

```text
3
```

---

## 9.5 DIVIDE

La fonction `DIVIDE` sert à faire une division.

On pourrait utiliser le symbole `/`, mais `DIVIDE` est souvent préférable, car elle gère mieux les divisions par zéro.

Exemple :

```DAX
TauxMarge =
DIVIDE(
    [Marge],
    [ChiffreAffaires]
)
```

Si le chiffre d’affaires est égal à zéro, une division classique peut provoquer une erreur ou un résultat problématique.

Avec `DIVIDE`, le calcul est plus propre.

On peut aussi ajouter une valeur de remplacement.

```DAX
TauxMarge =
DIVIDE(
    [Marge],
    [ChiffreAffaires],
    0
)
```

Cela veut dire :

> Si la division est impossible, retourne 0.

---

## 9.6 SUMX

La fonction `SUMX` est très importante.

Elle permet de faire un calcul ligne par ligne, puis d’additionner les résultats.

Exemple :

```DAX
ChiffreAffaires =
SUMX(
    Ventes,
    Ventes[Quantité] * Ventes[PrixUnitaire]
)
```

Cette formule veut dire :

1. Va dans la table `Ventes`.
2. Pour chaque ligne, calcule `Quantité × PrixUnitaire`.
3. Additionne tous les résultats.

### Différence entre SUM et SUMX

`SUM` additionne une colonne qui existe déjà.

`SUMX` calcule d’abord une expression pour chaque ligne, puis additionne.

### Exemple

| Produit | Quantité | PrixUnitaire |
| ------- | -------: | -----------: |
| Souris  |        3 |           20 |
| Clavier |        2 |           50 |
| Écran   |        1 |          200 |

Calcul :

```text
Souris  = 3 × 20  = 60
Clavier = 2 × 50  = 100
Écran   = 1 × 200 = 200
Total   = 360
```

Donc `SUMX` retourne :

```text
360
```

---

## 9.7 CALCULATE

La fonction `CALCULATE` est une des fonctions les plus importantes de DAX.

Elle permet de modifier le contexte de filtre.

Exemple :

```DAX
VentesCanada =
CALCULATE(
    [TotalVentes],
    Clients[Pays] = "Canada"
)
```

Cette formule veut dire :

> Calcule le total des ventes, mais en appliquant le filtre Pays = Canada.

Autre exemple :

```DAX
Ventes2025 =
CALCULATE(
    [TotalVentes],
    Date[Annee] = 2025
)
```

Cette formule veut dire :

> Calcule le total des ventes, mais seulement pour l’année 2025.

### Vulgarisation

`CALCULATE`, c’est comme donner une instruction spéciale à Power BI.

On lui dit :

> Fais le calcul habituel, mais change les conditions du calcul.

C’est une fonction fondamentale, car beaucoup d’analyses professionnelles nécessitent de changer ou de contrôler le contexte.

---

## 9.8 ALL

La fonction `ALL` permet d’ignorer certains filtres.

Elle est souvent utilisée pour calculer des pourcentages du total.

Exemple :

```DAX
PourcentageTotal =
DIVIDE(
    [TotalVentes],
    CALCULATE(
        [TotalVentes],
        ALL(Produits[Catégorie])
    )
)
```

Cette formule veut dire :

> Divise les ventes de la catégorie actuelle par les ventes de toutes les catégories.

### Vulgarisation

`ALL`, c’est comme dire :

> Pour ce calcul, oublie le filtre actuel et regarde le total complet.

Cela permet de comparer une partie avec le tout.

---

<details>
<summary>10 - Exemple complet : ventes, coûts, marge et pourcentage</summary>

<br/>

Dans cette section, on va prendre un exemple complet et progressif.

Imaginons une table `Ventes`.

| VenteID | Date       | Produit | Région   | ClientID | Quantité | PrixUnitaire | CoutUnitaire |
| ------- | ---------- | ------- | -------- | -------- | -------: | -----------: | -----------: |
| 1       | 2025-01-01 | Souris  | Montréal | C001     |        3 |           20 |           10 |
| 2       | 2025-01-02 | Clavier | Québec   | C002     |        2 |           50 |           30 |
| 3       | 2025-01-03 | Écran   | Montréal | C001     |        1 |          200 |          150 |
| 4       | 2025-01-04 | Souris  | Laval    | C003     |        5 |           20 |           10 |

---

## 10.1 Calculer le chiffre d’affaires

Le chiffre d’affaires correspond au montant total généré par les ventes.

Pour chaque ligne :

```text
Chiffre d’affaires ligne = Quantité × Prix unitaire
```

En DAX :

```DAX
ChiffreAffaires =
SUMX(
    Ventes,
    Ventes[Quantité] * Ventes[PrixUnitaire]
)
```

Explication :

Cette mesure parcourt chaque ligne de la table `Ventes`.

Pour chaque ligne, elle multiplie la quantité par le prix unitaire.

Ensuite, elle additionne tous les montants calculés.

---

## 10.2 Calculer le coût total

Le coût total correspond au coût supporté par l’entreprise pour les produits vendus.

Pour chaque ligne :

```text
Coût ligne = Quantité × Coût unitaire
```

En DAX :

```DAX
CoutTotal =
SUMX(
    Ventes,
    Ventes[Quantité] * Ventes[CoutUnitaire]
)
```

Explication :

Cette mesure calcule le coût de chaque vente, puis additionne tous les coûts.

---

## 10.3 Calculer la marge

La marge correspond à ce qui reste après avoir retiré les coûts du chiffre d’affaires.

```text
Marge = Chiffre d’affaires - Coût total
```

En DAX :

```DAX
Marge =
[ChiffreAffaires] - [CoutTotal]
```

Explication :

Cette mesure utilise deux mesures déjà créées.

Elle prend le chiffre d’affaires total et retire le coût total.

---

## 10.4 Calculer le taux de marge

Le taux de marge permet de comprendre la rentabilité en pourcentage.

```text
Taux de marge = Marge / Chiffre d’affaires
```

En DAX :

```DAX
TauxMarge =
DIVIDE(
    [Marge],
    [ChiffreAffaires],
    0
)
```

Explication :

Si le chiffre d’affaires est 0, la fonction retourne 0 au lieu de provoquer une erreur.

Le taux de marge est très utile, car il permet de comparer des produits ou des régions même si leurs ventes ne sont pas identiques.

Un produit peut avoir beaucoup de ventes, mais une marge faible.

Un autre produit peut avoir moins de ventes, mais une marge plus intéressante.

---

## 10.5 Calculer le nombre de clients distincts

Un même client peut faire plusieurs achats.

Il ne faut donc pas toujours compter les lignes de vente.

Il faut compter les clients uniques.

```DAX
NombreClients =
DISTINCTCOUNT(Ventes[ClientID])
```

Explication :

Cette mesure compte chaque client une seule fois, même s’il apparaît plusieurs fois dans la table.

---

## 10.6 Calculer le panier moyen

Le panier moyen indique combien un client dépense en moyenne.

```DAX
PanierMoyen =
DIVIDE(
    [ChiffreAffaires],
    [NombreClients],
    0
)
```

Explication :

Cette mesure divise le chiffre d’affaires par le nombre de clients distincts.

Elle permet de comprendre la valeur moyenne générée par client.

---

## 10.7 Calculer les ventes de Montréal

```DAX
VentesMontreal =
CALCULATE(
    [ChiffreAffaires],
    Ventes[Région] = "Montréal"
)
```

Explication :

Cette mesure calcule le chiffre d’affaires, mais seulement pour les lignes où la région est Montréal.

---

## 10.8 Calculer la part de chaque région dans le total

```DAX
PartRegion =
DIVIDE(
    [ChiffreAffaires],
    CALCULATE(
        [ChiffreAffaires],
        ALL(Ventes[Région])
    ),
    0
)
```

Explication :

Le numérateur représente les ventes de la région actuellement affichée.

Le dénominateur représente les ventes de toutes les régions.

Cette mesure permet donc de savoir quelle part du total est représentée par chaque région.

Exemple :

| Région   | ChiffreAffaires | PartRegion |
| -------- | --------------: | ---------: |
| Montréal |             260 |     56,5 % |
| Québec   |             100 |     21,7 % |
| Laval    |             100 |     21,7 % |

---

## 10.9 Pourquoi cet exemple est important ?

Cet exemple montre que DAX permet de construire une vraie analyse.

On ne se contente plus de dire :

> Voici les ventes.

On peut maintenant dire :

> Voici les ventes, les coûts, la marge, le taux de marge, le nombre de clients, le panier moyen et la contribution de chaque région au total.

C’est exactement la différence entre un rapport simple et un rapport analytique.

<details>
<summary>11 - Erreurs fréquentes des débutants</summary>

<br/>

Quand on commence avec DAX, certaines erreurs reviennent souvent.

Les comprendre permet d’éviter beaucoup de confusion.

---

## 11.1 Confondre Power Query et DAX

Power Query sert à préparer les données.

DAX sert à calculer des indicateurs.

Si une transformation doit nettoyer les données avant l’analyse, Power Query est souvent le bon endroit.

Si le calcul doit réagir aux filtres du rapport, DAX est souvent le bon choix.

---

## 11.2 Créer trop de colonnes calculées

Beaucoup de débutants créent des colonnes calculées pour tout.

Mais dans Power BI, les mesures sont souvent préférables pour les indicateurs.

Par exemple, pour calculer le total des ventes, il vaut mieux créer une mesure :

```DAX
TotalVentes = SUM(Ventes[Montant])
```

plutôt que de multiplier inutilement les colonnes calculées.

---

## 11.3 Ne pas comprendre pourquoi une mesure change

Une mesure change selon le contexte.

Ce n’est pas une erreur.

C’est justement la force de DAX.

Si une mesure donne un résultat différent dans une carte, un tableau ou un graphique, il faut regarder les filtres actifs.

---

## 11.4 Utiliser `/` au lieu de `DIVIDE`

Dans les calculs de pourcentage, il est préférable d’utiliser `DIVIDE`.

Exemple recommandé :

```DAX
TauxMarge =
DIVIDE(
    [Marge],
    [ChiffreAffaires],
    0
)
```

Cela évite les problèmes lorsque le dénominateur est égal à zéro.

---

## 11.5 Vouloir apprendre trop de fonctions trop vite

DAX peut sembler compliqué parce qu’il existe beaucoup de fonctions.

Mais au début, il suffit de bien comprendre quelques fonctions :

* `SUM`
* `AVERAGE`
* `COUNT`
* `DISTINCTCOUNT`
* `DIVIDE`
* `SUMX`
* `CALCULATE`
* `ALL`

Il vaut mieux maîtriser ces fonctions avec de bons exemples que mémoriser beaucoup de fonctions sans comprendre le contexte.

<details>
<summary>12 - Résumé final</summary>

<br/>

DAX est le langage de calcul de Power BI.

Il sert à créer des indicateurs qui n’existent pas directement dans les données brutes.

Power BI sans DAX permet de créer des rapports simples.

Power BI avec DAX permet de créer des rapports analytiques et professionnels.

La différence est importante.

Sans DAX, on affiche surtout les données.

Avec DAX, on analyse les données.

---

## Phrase à retenir

```text
Power Query prépare les données.
Le modèle de données organise les tables.
DAX calcule les indicateurs.
Power BI affiche les résultats.
```

---

## Autre phrase à retenir

```text
Les données brutes montrent ce qui existe.
DAX permet de comprendre ce que cela signifie.
```

---

## Résumé des rôles

| Élément           | Rôle                       |
| ----------------- | -------------------------- |
| Données           | Matière première           |
| Power Query       | Nettoyage et préparation   |
| Modèle de données | Relations entre les tables |
| DAX               | Calculs et indicateurs     |
| Visualisations    | Présentation des résultats |

---

## Ce que DAX permet de faire

DAX permet de calculer :

* un chiffre d’affaires ;
* une marge ;
* un taux de marge ;
* un pourcentage du total ;
* un panier moyen ;
* un nombre de clients distincts ;
* une croissance ;
* une comparaison avec l’année précédente ;
* un écart entre objectif et résultat ;
* un classement ;
* un indicateur de performance.

---

## Conclusion

DAX n’est pas seulement un langage de formule.

C’est le cœur analytique de Power BI.

Il permet de passer d’un rapport qui affiche simplement des données à un rapport qui aide réellement à comprendre une situation.

Dans un contexte professionnel, cette différence est essentielle.

Un bon rapport Power BI ne se limite pas à montrer des tableaux et des graphiques.

Il doit répondre à des questions.

Il doit expliquer une situation.

Il doit aider à prendre une décision.

DAX est l’outil qui permet de créer cette intelligence analytique dans Power BI.
